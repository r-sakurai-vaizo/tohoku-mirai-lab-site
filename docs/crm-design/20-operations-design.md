# 20. 運用設計書

## 運用体制

| 役割 | 担当 |
|---|---|
| システム管理者 | Admin |
| 編集責任者 | Reviewer |
| 記事作成 | Editor |
| 素材登録 | Contributor |
| 障害一次対応 | Admin |

## 日常運用

### 活動登録

1. 活動発生
2. ContributorまたはEditorが下書き登録
3. 素材を登録
4. 記録化が必要なら記録を作成
5. Editorが本文を整える
6. Reviewerへレビュー依頼

### 公開運用

1. Reviewerがチェックリスト確認
2. 必要なら差し戻し
3. 問題なければ承認
4. 公開予約または即時公開
5. 公開サイトで表示確認

### 素材運用

- 画像はalt、クレジット、利用範囲を必須入力
- 不明素材はINTERNAL_ONLY
- 公開後に権利問題が出た場合は即時非公開化

## 定期運用

| 頻度 | 作業 |
|---|---|
| 毎週 | レビュー待ち確認、下書き棚卸し |
| 毎月 | 公開記録の一覧確認、素材利用範囲確認 |
| 四半期 | 権限棚卸し、不要ユーザー無効化 |
| 半年 | DBバックアップ復元テスト |

## 障害対応

### 公開サイト障害

1. 障害範囲確認
2. 直近デプロイ確認
3. 必要ならロールバック
4. 原因と対応を運用ログへ記録

### CRM障害

1. ログイン可否確認
2. APIエラー確認
3. DB接続確認
4. Storage接続確認
5. 復旧まで公開サイトは既存公開データで維持

### 誤公開

1. Adminが対象をUNPUBLISHEDへ変更
2. 公開サイト反映を確認
3. 監査ログと差分を確認
4. 関係者へ状況共有
5. 修正版を作成し再レビュー

## バックアップ・復旧

- DBは日次バックアップ
- Storageは削除保護
- 月1回は復元手順を確認
- 重要公開前に手動バックアップ

## 権限棚卸し

- 3か月ごとにユーザー一覧を確認
- 活動していないメンバーはDISABLED
- Adminは最小人数にする
- 権限変更は理由を残す

## リリース運用

1. stagingへデプロイ
2. smoke test
3. DB migration確認
4. productionへデプロイ
5. 公開サイト主要ページ確認
6. エラーログ確認

## 運用上の禁止事項

- 認証なしでCRMを公開しない
- 下書きURLを外部共有しない
- 個人情報を公開本文に入れない
- 利用範囲不明の画像を公開しない
- 公開済み本文を承認なしで大幅変更しない
