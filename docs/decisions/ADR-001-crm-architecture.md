# ADR-001: 公開サイトとCRMを分離する

## Status

Accepted

## Date

2026-07-05

## Context

東北大未来探求ラボの公開サイトは、営業感を出さず、団体の社会的意義と記録を静かに見せる必要がある
一方で運営側では、活動、記録、素材、レビュー、公開承認、監査ログを扱うリッチなCRMが必要になる
公開サイトとCRMを同じ情報構造で混ぜると、内輪の運用情報が表に出やすくなる

## Decision

公開サイトとCRMを分離する
公開サイトは `団体について / 活動 / 記録` に絞り、CRMは認証付き管理画面として別領域で構築する
データはPostgreSQLを中心に管理し、公開APIは `PUBLISHED` のみ返す

## Alternatives Considered

### 静的JSON運用を継続

- Pros: 最速で運用開始できる
- Cons: 承認、権限、監査ログ、素材管理が弱い
- Rejected: リッチCRM要件に耐えない

### Headless CMSをそのまま使う

- Pros: 記事管理は速い
- Cons: 活動、素材、レビュー、権限、監査ログの独自要件が残る
- Rejected: 中核業務がCMSの制約に引っ張られる

### 公開サイトに管理機能を内包

- Pros: 小規模には単純
- Cons: 認証境界が曖昧になり、内輪情報が表に出るリスクがある
- Rejected: 公開サイトの思想とCRMの業務要件が違う

## Consequences

- 公開サイトは軽く、静かなトーンを維持できる
- CRMは認証、権限、承認、監査ログを前提に設計できる
- 初期実装は静的サイトより重くなる
- DB、認証、Storage、運用監視が必要になる
